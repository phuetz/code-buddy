# V1.0.0 Validation Campaign — 2026-06-10

Deep post-release validation of the `v1.0.0` GA tag, run on Ministar
Ubuntu (Ryzen AI 9 HX 470, Node v24.14.1, Ollama/Vulkan local models).
Every gate below was run against the tagged release plus the campaign's
own fixes (commits `ae98e424..`).

## Verdict

**GA confirmed.** Every release-critical surface validated end-to-end.
The campaign also caught and fixed **one real product bug** (MCP stderr
leak), **one operational gotcha** (Ollama `num_ctx`), and **22 stale
tests** that were masking themselves as environment noise.

## Findings

### F1 — MCP server stderr leaked into the CLI's stderr (product bug, FIXED)

`StdioTransport` created the SDK transport without `stderr`, whose
default is `'inherit'`: any noisy MCP server (e.g. the PdfCommander
banner) wrote raw bytes into buddy's stderr regardless of `--quiet` /
`LOG_LEVEL`, breaking the pipeable-headless contract and 8 tests.
**Fixed**: `stderr: 'pipe'` + drain to `logger.debug` (drain is
load-bearing — an unconsumed PassThrough back-pressures the child).

### F2 — Ollama default `num_ctx` silently kills the agentic loop (operational)

With Ollama's default context (4096) and Code Buddy's agentic system
prompt (~6k tokens), generation is truncated at 0 tokens
(`finish_reason: "length"`) and the turn degrades to the placeholder
"Using tools to help you...". Wire-level capture confirmed tools were
sent and the model never got room to answer. **Workarounds validated**:
derived model with `PARAMETER num_ctx 32768`, or
`OLLAMA_CONTEXT_LENGTH=16384+` on the service.
**V1.1 recommendation**: detect `finish_reason==='length'` with empty
content on round 0 and emit an actionable error naming `num_ctx`.

Also note: `gemma4*`/`llama3*`/`deepseek*`/`qwen2.5*` are deliberately
`supportsToolCalls: false` in `model-tools.ts` (chat-only); `qwen3*` is
the supported local agentic family.

### F3 — 22 of the 26 "environmental" failures were stale tests (FIXED)

The baseline cross-check (same failures on the parent commit) proved
them pre-existing, but deeper triage showed most pinned superseded
product states: docs inventories after the rebranding (9), the OpenClaw
WS fixture predating the a7354b11 paired-device handshake and 2026.6.x
`node.pair.*` method names (6), Cowork demo/skills/parity evolution
(5), and 2 fixed by F1. All realigned — see commits.

### F4 — Remaining 9 failures: genuinely machine-coupled (OPEN, non-blocking)

All reproduce in isolation on this box and predate the tag:

| File | Cause on this machine |
|:-----|:----------------------|
| `sandbox/os-sandbox` (2) | kernel supports landlock → backend recommendation differs from the pinned `bubblewrap` |
| `unit/background-task-manager` (1) | expects shell `sh`, machine resolves `bash` |
| `hooks/user-hooks` (1) | timeout-path test exceeds vitest's 5 s budget under tsx (+EPIPE) |
| `tools/ocr-tool` (1) | OCR batch > 20 s (engine slow/missing here) |
| `features/cloud-lsp-ide` (1) | expects an already-running LSP server state |
| `tools/computer-control-semantic-actions` (1) | needs a real window manager/display |
| `fleet/fleet-loopback-smoke` (1) | machine model config leaks into the routing recommendation |
| `sidecar/sidecar-bridge` (1) | real sidecar binary present but handshake times out |

**V1.1 recommendation**: pin env in these tests or add `skipIf` guards
(pattern: `tests/database.test.ts` native-module probe) so `npm test`
is 100 % green on any machine.

### F5 — Cowork `-real` bridge tests are load-sensitive (OPEN, minor)

`hermes-*-bridge-real.test.ts` boot the compiled core with fixed
timeouts; under full-suite parallelism 1–2 of them can flake. Green in
isolation. Consider serializing them or raising their timeouts.

```bash
cd cowork && npm test                 # cowork suite
npm run check:circular && npm pack --dry-run
npm run build
node dist/index.js server --port 3979 &  # then GET /api/health
CODEBUDDY_PROVIDER=ollama GROK_MODEL=qwen3.5-ctx32k \
  node dist/index.js -p "Read package.json and give the version" -o json
```
